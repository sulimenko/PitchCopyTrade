## P0 — Блокеры (исправить до любого merge)

### Безопасность
- [ ] Нет интерполяции строк в SQL — только SQLAlchemy bound params
- [ ] HMAC-SHA256 Telegram проверяется на каждом auth callback (`auth_date` < 5 мин)
- [ ] Подпись `initData` Telegram Mini App проверяется перед доверием identity
- [ ] `X-Internal-Token` проверяется на `/internal/broadcast` — жёсткая блокировка без него
- [ ] JWT: HttpOnly + Secure + SameSite=Strict, срок ≤ 24ч
- [ ] Никаких секретов (токенов, паролей, ключей) в коммитах и логах
- [ ] Автор читает/пишет только свои стратегии и рекомендации (ACL по `author_profile.id`)
- [ ] Маршруты `/admin/*` защищены `require_role("admin")`
- [ ] Подписчик не может зайти в кабинет автора или администратора

### Целостность данных
- [ ] `RecommendationLeg.instrument_id` получается по ticker — нет orphan-ног
- [ ] Переходы `Recommendation.status` только по разрешённому пути: draft → published → closed/cancelled
- [ ] `Payment.final_amount_rub` вычисляется на сервере, не принимается от клиента
- [ ] `Subscription` создаётся только после `paid` или ручного подтверждения админом

### Уведомления
- [ ] Постановка ARQ-задания — **fire-and-forget**: публикация не блокируется если worker упал
- [ ] Ошибки broadcast логируются, не проглатываются тихо
- [ ] ARQ `enqueue_job` вызывается атомарно с обновлением статуса рекомендации

## P1 — Высокий приоритет

### Роли и ACL
- [ ] 3 роли в БД: admin, author, moderator (moderator не используется в MVP)
- [ ] `requires_moderation=False` на всех AuthorProfile MVP — никакой маршрут не проверяет статус модерации
- [ ] Автор создаётся только админом — нет пути саморегистрации
- [ ] Уникальность `telegram_user_id` на уровне БД и приложения

### Логика рекомендаций
- [ ] Обязательные поля: только `ticker` + `side` при inline-создании
- [ ] Все ценовые поля (`entry_from`, `tp1`, `stop_loss`) nullable — пустое = NULL, не 0
- [ ] `RecommendationKind` по умолчанию `new_idea` при быстром inline-создании
- [ ] `published_at` — серверное UTC время, не клиентское
- [ ] Дублированная публикация предотвращена (статус должен быть `draft` для публикации)

### Инструменты
- [ ] Инструменты засеяны из `doc/instruments_stub.json` при старте (upsert по ticker)
- [ ] `GET /api/instruments` возвращает только `is_active=True`
- [ ] Поиск тикера нечувствителен к регистру
- [ ] Поля `last_price` и `change_pct` присутствуют в ответе (stub-значения OK)

### UI / Шаблоны
- [ ] Нигде нет онбординг-текста, инструкций, блоков подсказок
- [ ] Пустые таблицы показывают минимальный empty state («Нет рекомендаций»), не placeholder-строки
- [ ] Popup тикера закрывается по Esc и клику на backdrop
- [ ] Inline-строка сбрасывается после успешного сохранения
- [ ] HTMX-ответы используют правильные `HX-Trigger` или swap targets — нет полных перезагрузок страницы
- [ ] Весь пользовательский текст на русском

### База данных
- [ ] Новая Alembic-миграция для каждого изменения схемы
- [ ] Миграция обратима (`downgrade` реализован)
- [ ] Нет `CREATE TABLE` в коде приложения — только через миграции
- [ ] `Base.metadata` включает все модели до запуска `env.py`

## P2 — Стандартное качество

### Стиль кода
- [ ] Нет неиспользуемых импортов
- [ ] Нет закомментированных блоков кода
- [ ] Нет `print()` — только `logging`
- [ ] Функции до 50 строк (разбить большие)
- [ ] Нет magic strings — использовать enum из `db/models/enums.py`

### Дизайн API
- [ ] JSON API эндпоинты под префиксом `/api/`
- [ ] HTMX partial эндпоинты под `/cabinet/` или `/admin/`
- [ ] Ошибки возвращают правильные HTTP коды (400/401/403/404/422/500)
- [ ] Тело 422 содержит field-level ошибки валидации

### Конфигурация
- [ ] Все новые config-значения добавлены в `core/config.py` с типами и дефолтами
- [ ] Новые env-переменные задокументированы в README
- [ ] Нет захардкоженных URL, портов, секретов

### Тесты
- [ ] Каждый новый маршрут имеет хотя бы один тест
- [ ] Тест ACL: неавторизованный доступ возвращает 401/403
- [ ] Happy path тест: корректные данные возвращают ожидаемый ответ

## P3 — Проверка product drift

Проверяем что реализация не отходит от утверждённого blueprint:

- [ ] Кабинет автора — только веб, никаких bot-команд для авторов
- [ ] Флоу подписчика — только Telegram bot + Mini App, без отдельной веб-регистрации
- [ ] One Pager — HTML в `Strategy.full_description`, рендерится на сервере
- [ ] Подтверждение платежа — ручное (действие админа), без автоматических вызовов платёжного API
- [ ] Роль Модератора: enum есть, UI нет, проверок нет, `requires_moderation=False`
- [ ] MinIO: инфраструктура поднята, маршруты загрузки не реализованы в MVP
- [ ] Режим бота: `TELEGRAM_USE_WEBHOOK=true` в production, `false` в local dev
- [ ] Инструменты: только из `instruments_stub.json` — никаких live market API вызовов в MVP
